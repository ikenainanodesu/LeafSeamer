01 / Overview

项目概览

LeafSeamer 是 NodeCG 播控系统，使用可独立安装的 Bundle 组合连接节目流程、日志、排期、图文和外部音视频设备。

核心功能

Seamer 卡片与触发器、Logger、Schedule、OBS、ATEM、Mixer、VB Matrix、Graphics、Backup、Google Sheets 数据同步。

语言与前端

TypeScript / TSX / CSS / HTML，React 19 与 Vite 6 构建 Dashboard 和 Graphic。

后端与通信

Node.js、NodeCG extension、Replicant、listenFor 和跨 Bundle message。

架构

核心 Bundle 独立运行，可选 Adapter 负责 Seamer、Schedule 与设备或数据源之间的桥接。

构建 Workflow

npm install → npm run typecheck → npm run build → npm start。

关键工具

NodeCG 2.6.4、TypeScript 5.7、GSAP、obs-websocket-js、atem-connection、node-osc、googleapis、archiver。

结论：核心 Bundle 已具备独立加载方向；当前主要缺口集中在安全边界、敏感数据、自动化独立性验证、设备能力探测和真实硬件验收。

Platform / Shared concerns

平台共享

承载无法合理归属于单个业务 Bundle 的部署、安全、命令边界、构建和数据治理问题。

职责

NodeCG 运行入口、共享消息边界、构建编排和部署治理。

独立条件

每个 NodeCG 实例都必须具备安全入口、最小网络权限和可重复构建。

关键入口

cfg/nodecg.json、根脚本、各 Bundle listenFor 与 Replicant。

PLAT-01

NodeCG Dashboard 与硬件控制入口可能暴露

P0

现状证据

cfg/nodecg.json.example 使用 host: 0.0.0.0；系统可控制 OBS、ATEM、Mixer、VB Matrix 与备份。

风险

仅修改 host 只能限制 NodeCG 监听的网卡，不能限制可访问该网卡的来源客户端；任何可路由到目标地址的设备仍可能获得播控能力。

影响范围

所有 Dashboard、设备控制 Bundle、备份数据和现场播出。

短期缓解

单机绑定 127.0.0.1；局域网绑定播控网卡 IP，并用 Windows 防火墙限制来源网段。生产环境禁止直接暴露公网。

长期改善

跨网访问时由 Nginx、Caddy 或 Traefik 对外监听 443、终止 TLS、完成 OIDC/Basic Auth/访问码身份组认证，再转发到仅监听本机的 NodeCG。

实施路径

  1. 在 README/安装手册写明监听地址、防火墙、TLS、认证、Secret、日志、备份和恢复的最低部署检查表。
  2. 由可管理交换机划分播控 VLAN，并在路由器或三层交换机配置 VLAN 间 ACL；只有 VLAN、没有 ACL 不构成完整隔离。
  3. 建立设备端口和来源网段 allowlist。
  4. 增加只读访问模式。

验收标准

NodeCG 只监听预期网卡；非允许来源无法访问 Dashboard 或设备端口；跨网访问使用 HTTPS 和认证；只读身份不能执行命令。

PLAT-02

敏感数据缺少统一 Secret 边界

P0

现状证据

OBS stream key/password、Google credentials 和未来设备 token 可能进入普通配置、Replicant、日志或备份。

风险

敏感数据会被 Dashboard 读取、持久化到 db，或随日志和 zip 被复制。

影响范围

OBS、Data Sync、Logger、Backup，以及未来带密码的设备 Bundle。

短期缓解

禁止在日志中输出 secret；备份排除已知敏感路径；UI 只显示 configured 状态。

长期改善

实现可打包进每个 Bundle 的服务端 SecretManager Library。可逆 Secret 按模块 namespace 使用 AES-256-GCM 加密，master key 来自环境变量或系统 Secret Store；访问码只需验证，使用 Argon2id 或 bcrypt 单向哈希。

实施路径

  1. 建立 secret inventory 与模块 namespace。
  2. 从 Replicant 剥离明文，Dashboard 只接收 configured 和更新时间。
  3. 加入 AES-256-GCM、key version 与 rotation。
  4. 添加统一 redaction，并让 Backup 只能请求分类后的加密导出。

验收标准

Replicant、普通配置、日志和备份中搜索不到明文；前端只得到脱敏状态。

PLAT-03

控制消息缺少统一授权与 Payload 合同

P0

现状证据

Dashboard 可使用 Replicant 和 sendMessage；各 Bundle 的 listenFor 尚无统一 schema、权限和目标 allowlist。

风险

错误类型、越界值、未知连接或低权限客户端可能触发硬件命令或污染权威状态。

影响范围

所有设备 Bundle、Seamer、Adapter、备份和敏感设置操作。

短期缓解

优先为推流、Macro、VB patch、Backup 和 Secret 保存命令添加 validator 与目标检查。

长期改善

建立可打包的 CommandGateway 中间件：每个 Bundle 自己注册命令 handler，统一执行身份 → 权限 → payload schema → 目标 allowlist → 执行 → 脱敏审计；不引入必须安装的中央 NodeCG Bundle。

实施路径

  1. 生成 message registry。
  2. 定义带版本、身份上下文和 correlationId 的 Command Envelope/Ack。
  3. 优先迁移 OBS 推流、ATEM Macro、VB Patch、Backup 和 Secret 保存。
  4. 再迁移其余入口并加入权限矩阵。

验收标准

非法、越界、未知目标和无权限消息全部被拒绝；合法消息返回统一 ack 并留下脱敏审计。

PLAT-04

Bundle 独立性缺少自动化构建与安装验证

P1

现状证据

项目已有 npm workspaces 和单 Bundle build 入口，但报告基线没有 CI、依赖审计或独立安装矩阵结果。

风险

隐藏 import、共享路径或构建脚本回归可能让某个 Bundle 必须依赖完整仓库才能工作。

影响范围

所有核心 Bundle 和 Adapter 的独立部署承诺。

短期缓解

在提交前运行根 typecheck、全量 build 和关键 Bundle 单独 build。

长期改善

建立 CI matrix：最小 NodeCG + 单核心 Bundle、核心 + 对应 Adapter、完整组合。

实施路径

  1. 列出支持组合。
  2. 定义 smoke test。
  3. 加入依赖审计。
  4. 保存构建产物检查。

验收标准

每个核心 Bundle 在不安装 Seamer 和无关设备 Bundle 时可构建、加载并打开 Dashboard。

PLAT-05

Vite 源码目录与输出目录边界不清晰

P1

现状证据

独立 Bundle 的 Vite 配置以 src 为 root，却输出到 Bundle 运行目录，已出现 outDir/root 父子关系警告。

风险

清理产物时可能污染源码，构建结果与 NodeCG 运行目录职责混杂。

影响范围

使用 Vite 的 Dashboard 与 Graphic Bundle。

短期缓解

构建前检查目标路径，不对父级目录执行宽泛清理。

长期改善

源码只放 src，先输出到临时目录，再由构建脚本复制到 NodeCG 运行目录。

实施路径

  1. 选择一个 Bundle 试点。
  2. 拆分临时输出。
  3. 更新 fix paths。
  4. 逐 Bundle 迁移。

验收标准

全量构建无 outDir/root 警告；清理产物不会改动源码；NodeCG 仍能加载输出。

PLAT-06

实时状态、事件和历史数据缺少统一治理规则

P2

现状证据

当前主要依赖 Replicant 与 message；设备状态、审计、同步记录和高频指标的数据生命周期不同。

风险

高频数据无限写入 Replicant 会增加同步压力；审计和历史记录又可能随普通日志清理丢失。

影响范围

Logger、OBS 统计、VB Matrix、未来 Mixer meter、Data Sync 和访问 session。

短期缓解

Replicant 仅保留低频配置和当前快照；高频状态先节流成 summary。

长期改善

建立明确分工：Replicant 保存“当前是什么状态”并同步 UI；版本化 Event Envelope 传递“刚发生什么变化”；SQLite + WAL 保存“以前发生过什么”，用于审计、同步历史、查询和恢复。WAL 改善并发与崩溃恢复，但不提供加密。

实施路径

  1. 分类现有数据。
  2. 定义保留周期。
  3. 先迁移审计和同步历史。
  4. 按需要引入聚合。

验收标准

每类数据都有 owner、通道、保留周期和恢复方式；SQL 不承担实时 UI 分发。

独立状态

SEAMER-01

固定四模块类型阻止新 Adapter 真正即插即用

P1

现状证据

Integration Registry 能自动显示已注册 Provider，但 TriggerModule、Trigger Manager 的条件分支和 EditTriggerModal 仍硬编码 mixer/atem/obs/vb。

风险

未来新增 Schedule、NDI 或其他模块时仍需修改 Seamer 核心类型、判断逻辑和 UI，独立 Adapter 只能解决加载依赖，不能解决功能扩展。

影响范围

Seamer、所有第三方 Adapter、触发器持久化兼容和独立部署。

短期缓解

继续禁止 Seamer 直接 import 设备 Bundle；新能力只能通过 Adapter 注册，缺失 Adapter 时配置标记 unavailable。

长期改善

Adapter 注册版本化 Capability Manifest，声明 actions、triggers、参数 schema、UI hints、状态快照、条件比较器和执行器；Seamer 根据 Manifest 动态生成表单。

实施路径

  1. 建立 Integration SDK 与 Manifest schema。
  2. 让 Registry 接受任意命名空间 ID。
  3. 把固定条件检查迁到 Provider evaluator。
  4. 把 Trigger UI 改为 schema 驱动。
  5. 把“仅安装 Seamer”与动态测试 Adapter 加入 CI。

验收标准

测试 Adapter 无需修改 Seamer 核心即可出现触发器和动作；卸载后配置保留但不可执行；Seamer 单独安装仍可加载。

PLAT-04、ADAPTER-01

Service bundle

Logger System

集中接收 Bundle 日志，支持 Log Viewer、Bundle 筛选、磁盘持久化与用户可选自动清理周期。

独立状态

Independent

入口

可选日志捕获、日志 Replicant、文件存储和清理任务。

依赖

其他 Bundle 接入可选；缺失时 Logger 仍可工作。

LOGGER-01

集中日志可能持久化敏感运行信息

P1

现状证据

Logger 会采集各 Bundle 服务端日志并写入磁盘；当前 record() 接收最终字符串，没有结构化敏感字段清洗。

风险

错误堆栈、设备地址、配置路径、Bearer Token 或凭据片段可能进入文件和 Viewer。

影响范围

所有接入日志捕获的 Bundle、Log Viewer 用户和日志目录备份。

短期缓解

禁止主动记录 secret；降低生产 debug 级别；限制日志目录权限。

长期改善

写入前先按字段名清洗 password/token/secret/key 等结构化数据，再检测 Bearer Token、私钥和带凭据 URL 等字符串模式。当前没有日志导出功能，因此“导出前二次脱敏”只作为未来导出 API 的强制边界，不提前增加无调用功能。

实施路径

  1. 建立敏感字段词典。
  2. 新增 sanitizeLogEntry 并在 record() 前执行。
  3. 添加测试 secret 与误报测试。
  4. 未来新增导出时复用同一 sanitizer。

验收标准

测试 secret 不出现在 Replicant、日志文件或 Viewer；普通场景名和设备标签不被误清洗。

关联问题

PLAT-02

LOGGER-02

普通日志清理策略不应承担安全审计生命周期

P1

现状证据

Log Viewer 已允许用户选择自动清理周期；未来命令授权方案需要不可被普通日志策略误删的审计记录。

风险

危险操作的身份组、目标和结果可能随普通运行日志一起删除，无法复盘。

影响范围

Command Gateway、设备命令、Secret 修改、Backup 和访问管理。

短期缓解

将关键操作写入独立文件，并使用固定最小保留期。

长期改善

建立独立 Audit Store：SQLite 保存可查询的追加记录，WAL 允许写入时继续读取并改善崩溃恢复；它不负责加密。字段包含时间、身份组、命令、目标、结果和 correlationId，参数必须先脱敏。

实施路径

  1. 定义 audit schema 和独立保留周期。
  2. 与普通 logger 分流。
  3. 由 Command Gateway 写入统一 Audit Event。
  4. 提供只读查询并限制访问。

验收标准

执行普通日志清理后审计记录仍在；危险命令可按时间和 correlationId 追踪到请求、设备调用和结果；库中无明文 secret。

Core bundle

Schedule Manager

目标职责调整为标准化外部播单、版本与状态机，并向 Seamer 发布显式自动化事件；Google Sheets 和 PostgreSQL 通过独立 Node.js Adapter 接入。

独立状态

数据源

自动化

SCHED-01

Schedule 仍是简单列表，尚未具备多来源同步与 Seamer 自动化合同

P1

现状证据

当前 ScheduleItem 只有 id/time/title/description/active；Sheets Adapter 从固定 sheetData 直接替换完整列表，没有 PostgreSQL、版本、冲突、回滚或 Seamer 事件。

风险

列结构变化、本地编辑覆盖或任意数据变化都可能造成播单错乱；若直接把所有变化当触发器，还可能误触发播出。

影响范围

Schedule Manager、Schedule Graphic、Google/PostgreSQL Adapter 和 Seamer 自动化。

短期缓解

导入前校验必需列，并记录来源、版本、时间和行数；任意新增行或任意字段变化不默认触发动作。

长期改善

定义标准 PlaylistItem 与版本化 Import Batch，支持 dry-run、diff、冲突和回滚；数据源 Adapter 只负责转换，Schedule 负责提交；Seamer Schedule Adapter 只发布 schedule.item_due 和配置的 schedule.field_changed。

实施路径

  1. 定义 sourceId/externalId/revision/plannedAt/state/metadata/triggerMappings。
  2. 重构 Google Sheets Adapter。
  3. 新增 PostgreSQL Adapter。
  4. 新增 Seamer Schedule Adapter 与动态 Manifest。
  5. 保存同步版本和回滚点。

验收标准

Google Sheets 与 PostgreSQL 生成同一数据模型；缺列和冲突不会覆盖当前播单；只有显式到期或字段迁移规则触发 Seamer。

VB-01

大矩阵刷新可能产生高密度 VBAN-TEXT 请求

P1

现状证据

扫描会对已发现 input/output 组合发送 dBGain 与 Mute 查询。

风险

影响范围

长期改善

加入节流、分批、差分、请求去重、可见区域扫描和失败退避。

实施路径

  1. 测量当前请求数。
  2. 实现队列与限速。
  3. 缓存稳定 slot。
  4. 按变更差分刷新。

验收标准

规定规模矩阵的峰值请求率受控；连续刷新不阻塞 patch；超时 slot 自动退避。

PLAT-06

VB-02

官方 Slot 发现与节点控制仍缺少真实设备端到端验收

P1

现状证据

Windows/VAIO/VBAN/VASIO 发现路径与 Point SUID 已按官方资料修正，但开发日志明确尚未在真实 Matrix 实例完成全链路验证。

风险

不同版本的返回格式、slot 数量或命令行为可能让设备缺失、状态误判或 patch 失败。

影响范围

设备发现、Matrix View、Patch Control 和 preset 恢复。

短期缓解

保留 Patch Control 作为已知路径，并在日志中记录原始设备响应。

长期改善

建立按 Matrix 版本和 slot 类型划分的真实设备验收记录与 parser fixture。

实施路径

  1. 采集 Slot(SUID).Info 返回。
  2. 验证发现矩阵。
  3. 逐类 patch/de-patch。

PLAT-03、SEAMER-01

Approved direction

推荐目标架构

采用“模块化 SDK + 可选 Adapter”，不采用中央强依赖 Bundle，也暂不采用 Python sidecar。

1. Source Adapters

Google Sheets 与 PostgreSQL Node Adapter 读取外部数据并输出版本化 Import Batch。

2. Schedule Core

执行 schema 校验、dry-run、diff、冲突处理、提交和回滚，发布当前播单快照。

3. Event Envelope

只发布 schedule.item_due 与配置的 schedule.field_changed，不把任意数据变化当播出命令。

4. Seamer Capability

Seamer Schedule Adapter 注册 Manifest，Seamer 自动生成触发器配置并保持核心无 Schedule 依赖。

5. Command Gateway

动作经过身份、权限、schema、allowlist、执行和脱敏审计。

6. Persistence

Replicant 管当前状态，Event 管变化，SQLite + WAL 管审计、同步历史和查询。
